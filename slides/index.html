The Problem: Agents Have Amnesia

Every Session Starts From Scratch

The agent reads the same files, hits the same dead ends, and rediscovers the same architectural quirks — over and over

No Cross-Session Learning

Last week's debugging breakthrough is locked in a JSONL transcript that no agent will ever read again

No Cross-Developer Transfer

Alice's agent learns a pitfall on Tuesday. Bob's agent rediscovers it on Thursday. The team pays twice

Onboarding Is Manual

New developers spend weeks absorbing tribal knowledge that agents could have accumulated automatically

The Reasoning DAG

Each session becomes a directed graph of typed reasoning nodes with causal edges:

HYPOTHESIS

Agent forms a theory

INVESTIGATION

Examines evidence

DISCOVERY

Finds the unexpected

PIVOT

Changes approach

DEAD END

Path that failed

SOLUTION

Working resolution

Edges: led_to, refined, contradicted, caused_pivot_to, resolved. Pivots and dead ends are first-class — often the most valuable signals.

6 / 16

Token-Budget Windowing (Cold Tier)

Before: Fixed 5-Message Windows

Long sessions → 30+ tiny windows. The LLM never saw enough context. Reasoning split mid-thought.

After: Token-Budget Windows

Same session → 2–3 large windows, each at 45% of 200K context. Single-window fast path for short sessions.

Why 45%?

Chroma 2025

All 18 frontier models degrade with increasing input length

Liu et al. 2024

"Lost in the Middle" — 30%+ accuracy drops for mid-context information

Paulsen 2025 (MECW)

Effective context is drastically below advertised maximums

Knowledge Consolidation

Raw reasoning nodes are clustered and classified into three knowledge types:

AI

Architectural Insights

Structural understanding of the codebase. Component relationships, configuration patterns, dependency rules.

!

Known Pitfalls

Recurring problems ranked by severity. Each includes a resolution strategy derived from how past agents solved them.

Dx

Diagnostic Strategies

Proven debugging approaches with ordered steps and success rates from prior sessions.

Plus key patterns (recurring successes) and anti-patterns (approaches that consistently fail)

Project vs. Team Scope

Every memory entry is classified at profile-build time:

PROJECT Scope

"The supply-chain project's Alembic migrations require all models to be registered in models/__init__.py for autodetect to work."

Useless for any other repo.

TEAM Scope

"Pydantic v2 requires migrating from class Config to model_config = ConfigDict(...). Old patterns silently work but emit deprecation warnings."

Applies to every Python project using Pydantic.

Team-scope memories cross project boundaries automatically on cmm pull.

Human-in-the-Loop Review

Reviewers see full context: type, scope, confidence, source developer, session ID, evidence.

  • Approve — promotes to shared main
  • Reject — soft delete with audit reason
  • Swap scope — toggle project ↔ team
  • Edit summary — fix the text before approving

All actions logged to data/sync/sync.db for audit.

# cmm review --project supply-chain [3/12] HYPOTHESIS · scope=project confidence=0.78 · by alice session=abc123 Summary: The bug appears to be in the migration runner — Alembic isn't picking up new models. Evaluation: Four Helpfulness Signals

Signal A: Errors Resolved

DEAD_END → memory retrieval → SOLUTION within 8 messages

Signal B: Pitfalls Avoided

Surfaced pitfall has no matching DEAD_END (embedding cosine ≥ 0.70)

Signal C: Pivots After Retrieval

PIVOT within 5 messages of a /search-memory invocation

Signal D: Harmful Memory

Loaded memory semantically matches a subsequent DEAD_END — memory may have misled

Position estimation uses real JSONL message counts at invocation time, not even-distribution guesses

Summary

Cognitive Memory Manager

  • Token-budget extraction — 45% context fill ratio, 2–3 large windows instead of 30 tiny ones
  • Two extraction tiers — warm (free, <1s) for every session, cold (LLM) for on-demand upgrades
  • Project / team scope — team-level knowledge crosses repo boundaries automatically
  • Distributed store — local ChromaDB cache + shared team store on any filesystem
  • Human-gated review — push/review/pull with full audit trail, prevents institutional hallucination
  • Four evaluation signals — errors resolved, pitfalls avoided, pivots influenced, harmful memory tracked
  • One-command onboarding — cmm init --shared gives a new developer the full team brain in under a minute
