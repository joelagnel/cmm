The Problem

Coding agents have amnesia.
Teams have it twice.

Every Claude Code session starts from scratch. The agent reads the same files, hits the same dead ends, and rediscovers the same architectural quirks — over and over.


Worse: each developer's agent learns alone. Alice's debugging insight on Tuesday never reaches Bob on Thursday. The team rediscovers the same pitfalls in parallel.

  • No memory of past dead-ends or pivots
  • No accumulated knowledge across sessions
  • No cross-developer knowledge transfer
  • Same bug debugged 4 times by 4 developers
  • Onboarding a new dev takes weeks of tribal knowledge

Reasoning DAGs as institutional memory.

Phase 1 · Cold Tier

Token-budget extraction.

Fixed 5-message windows split reasoning sequences across boundaries arbitrarily. The new approach packs each LLM call to 45% of the model's context window using the Anthropic SDK's token counter — research-backed safe utilization.

Before

Fixed 5-message windows

Long sessions produced 30+ tiny windows. The LLM never saw enough context to identify cross-cutting patterns.

  • 30 windows × small context
  • One node per window
  • Reasoning split mid-thought
After

Token-budget windows

Same session → 2-3 large windows, each carrying as much context as the model can confidently process. Single-window fast path for sessions that fit.

  • ~85k token budget per window
  • Multiple nodes per window
  • 750-token trailing overlap
04 / 14
Phase 1 · The Research

Why 45%?

LLM performance degrades as context fills up — even within the advertised window. Our 45% target sits well below known degradation thresholds while maximizing how much session content the model can actually reason over.


CMM_CONTEXT_FILL_RATIO=0.45

Configurable per-project via env or config file.

Chroma 2025

Universal degradation

All 18 frontier models degrade as input length grows.

Liu et al. 2024

Lost in the Middle

30%+ accuracy drops for information mid-context.

Paulsen 2025 · MECW

Effective Context

Real usable window is drastically below the advertised maximum.

Phase 2 · Scope

Project-specific or team-wide?

Every memory is classified at profile-build time as either PROJECT (specific to one repo) or TEAM (general knowledge). Team-scoped insights cross project boundaries automatically.

PROJECT scope

This codebase only

"The supply-chain project's Alembic migrations require all models to be registered in models/__init__.py for autodetect to work."

Useless for any other repo.

TEAM scope

Any project, any team

"Pydantic v2 requires migrating from class Config to model_config = ConfigDict(...). Old patterns silently work but emit deprecation warnings."

Applies to every Python project using Pydantic.

09 / 14
Phase 5 · The Approval Gate

Why human review matters.

Without a gate, one agent's hallucinated insight becomes institutional false knowledge for the entire team. The reviewer is the firewall.


Reviewers see node type, scope, confidence, source developer, and the original session ID. They can approve, reject with a reason, swap project ↔ team scope, or edit the summary inline.


All actions logged to data/sync/sync.db — auditable, attributable, and rejection-reasons preserved for postmortems.

# cmm review --project supply-chain [3/12] HYPOTHESIS · scope=project confidence=0.78 · by alice session=abc123 Summary: The bug appears to be in the migration runner — Alembic isn't picking up new models. Evidence: ImportError raised at line 42 of alembic/env.py during autogen. [a]pprove [r]eject [s]wap-scope [e]dit [k]skip [q]uit › a ✓ Approved

Phase 6 · Onboarding

A new developer in one command.

Clone the repo, run one command, get the entire team's accumulated knowledge in your local cache. No tribal knowledge handoff. No "oh by the way, watch out for…" conversations.


The cmm init --shared command writes the config, runs cmm pull, and refreshes the cached profile — all in under a minute.


After init, every Claude Code session on this project automatically loads the team profile at startup via the session-start hook.

# Day 1 on a new team: $ git clone git@github.com:team/project $ cd project $ cmm init . \ --shared /shared/team/memory \ --developer bob \ --team-id logistics ✓ Initialized: project-a3f2b1 Mode: shared Pulling existing team memories... ✓ Pulled 34 project nodes + 15 team nodes ✓ Cached profile updated # That's it. Full team brain loaded.

Evaluation

Three signals for helpfulness.

Signal A

Errors Resolved

DEAD_END → memory retrieval → SOLUTION within 8 messages. Did memory help fix it?

Signal B

Pitfalls Avoided

Surfaced pitfall has no matching DEAD_END node in the session. Was the trap dodged?

Signal C

Pivots After Retrieval

PIVOT node within 5 messages of a search. Did memory change direction?

Sessions split by memory_used_at_start — compare dead ends, pivots, messages-to-solution between cohorts. Audit trail in SQLite tracks every push, pull, approve, reject.
